## 2. Назначение и цели создания системы

### 2.1 Назначение

GTCP MVP — специализированная информационная система класса ETRM для автоматизации коммерческой деятельности компании-оператора газовых транзитных мощностей на границах Сербии (точки **Хоргош** и **Господжинцы**, коридор Центральная Европа — Балканы).

Система охватывает полный операционный цикл: от регистрации заявок грузоотправителей на транзит и управления номинациями до выставления счётов, мониторинга кредитных лимитов и контроля баланса мощностей.

### 2.2 Стратегические цели

1. **Цифровизация** — перевод процессов из ручного (Excel/email) режима в автоматизированную платформу
2. **Управление рисками в реальном времени** — мониторинг кредитных лимитов, автоматические Margin Call
3. **Прозрачность и аудируемость** — полная история всех транзакций с привязкой к пользователям
4. **Интеграционный потенциал** — API-first архитектура, готовность к подключению к TSO-платформам и ERP
5. **Снижение операционных рисков** — исключение ошибок ручного ввода, автопроверки лимитов и балансов

### 2.3 KPI (As Is → To Be)

| KPI | Текущее (As Is) | Целевое (To Be) |
|---|---|---|
| Время обработки номинации | 30–60 мин (email) | < 5 мин |
| Выявление превышения лимита | 1–24 часа | < 1 мин |
| Ошибки при выставлении счётов | 3–5% | 0% |
| Время подготовки отчёта по балансу | 2–4 часа | Мгновенно |
| Охват автоматизированных процессов | ~20% | ≥ 80% (MVP) |

## 4. Требования к системе

### 4.1 Функциональные требования (FR)

#### FR-01 · Аутентификация и авторизация
Система обеспечивает вход пользователей по логину/паролю. Разграничение доступа по ролям (RBAC). Журналирование входов/выходов.

#### FR-02 · Dashboard
Система отображает агрегированные KPI:
- Суммарные номинированные объёмы (МВт·ч/сутки)
- Количество активных грузоотправителей
- Объём выставленных счётов (EUR)
- Статус кредитных лимитов и оперативные тревожные сигналы

#### FR-03 · Реестр грузоотправителей (Shippers)
Система хранит реестр активных грузоотправителей с атрибутами: наименование, договор, мощность (МВт/ч), кредитный лимит, текущая экспозиция, статус.

#### FR-04 · Создание номинации
Пользователь создаёт номинацию: грузоотправитель, точка входа/выхода, объём (МВт/ч), дата газовых суток. Система проверяет наличие свободной мощности и кредитный лимит.

#### FR-05 · Статусная модель номинации
Система поддерживает жизненный цикл:
```
PENDING → MATCHING → CONFIRMED
                   → REJECTED
CONFIRMED → RENOM → CONFIRMED (после одобрения)
```
Все переходы логируются с временной меткой и пользователем.

#### FR-06 · Автоматический матчинг
По запросу диспетчера система запускает алгоритм матчинга: сопоставление встречных номинаций ENTRY/EXIT одного грузоотправителя за те же газовые сутки.

#### FR-07 · Реноминации (RENOM)
Система поддерживает изменение объёма номинации в D-day:
- Изменение ≤ ±10% от базового объёма → автоматически
- Изменение > 110% → требуется одобрение диспетчера

#### FR-08 · Кредитный монитор
Система отображает таблицу кредитных позиций: лимит, Current Exposure, доступный остаток, процент использования. Цветовое кодирование: зелёный (<80%), жёлтый (80–100%), красный (>100%).

#### FR-09 · Margin Call
При превышении кредитного лимита — автоматический тревожный сигнал. Форма Margin Call с автозаполнением суммы превышения, срок исполнения, правовое основание. История MCL.

#### FR-10 · Баланс мощностей
Система отображает суточный баланс по точкам Хоргош/Господжинцы: физические мощности, номинированные объёмы, свободный остаток, имбаланс (ENTRY − EXIT).

#### FR-11 · Биллинг — создание счёта
Создание счёта-фактуры с автозаполнением: номер INV-ГГГГ-НННН, грузоотправитель, период, объём, тариф, автоматический расчёт суммы. Статусы: ВЫСТАВЛЕН → ОЖИДАЕТ ОПЛАТЫ → ОПЛАЧЕН / ПРОСРОЧЕННЫЙ.

#### FR-12 · Синхронизация с ERP
Функция инициирования синхронизации данных биллинга с внешней ERP-системой (1С). MVP: симуляция вызова с фиксацией события в журнале аудита.

#### FR-13 · Управление договорами
Реестр договоров: тип (транзит/транспортировка/рамочный), стороны, срок действия, объём мощностей, статус. Выделение истекающих (<30 дней) и просроченных.

#### FR-14 · Управление мощностями (Capacity)
Журнал бронирований: контрагент, точка, период, тип (FIRM/INT), статус. Фильтрация по точке и типу.

#### FR-15 · Журнал аудита
Все значимые действия фиксируются: кто, что, когда, модуль. Фильтрация по модулю (AUTH/NOM/CREDIT/BILLING/CONTRACTS/SYSTEM). Журнал не редактируется пользователями.

### 4.4 Требования к безопасности

| ID | Требование |
|---|---|
| **SEC-01** | Обязательная авторизация для всех модулей. Пароли — bcrypt/Argon2, минимум 8 символов |
| **SEC-02** | RBAC: принцип минимальных привилегий, пользователь видит только свои модули |
| **SEC-03** | Все соединения только по HTTPS (TLS 1.2+), HTTP → redirect на HTTPS |
| **SEC-04** | Защита от CSRF, XSS, SQL-инъекций. Валидация всех входных данных на сервере |
| **SEC-05** | Журнал аудита безопасности: входы, изменения лимитов, Margin Call, изменения счётов |
| **SEC-06** | Автозавершение сессии через 30 мин неактивности |

### 4.5 Требования к интерфейсу пользователя

- Поддерживаемые браузеры: Chrome 100+, Firefox 100+, Edge 100+
- Разрешение: 1280×768 и выше
- **Тёмная тема** (#0a0e14 background) — стандарт торговых терминалов
- Моноширинные шрифты для числовых данных (IBM Plex Mono)
- Язык интерфейса: русский + латинские аббревиатуры (ENTRY, EXIT, FIRM, INT)
- Время в UTC+1 (CET) — стандарт европейского газового рынка
- Цветовое кодирование статусов: `#00E5A0` (норма) · `#FFD166` (внимание) · `#FF4D6D` (нарушение)
- Модальные окна для форм (без перехода на новую страницу)
- Toast-уведомления о результатах операций (успех/ошибка/предупреждение)

## 6. Порядок контроля и приёмки

### 6.1 Definition of Done (MVP)

- [ ] Все 15 FR (FR-01–FR-15) реализованы и проверены
- [ ] Покрытие unit-тестами расчётного ядра ≥ 80%
- [ ] Все 6 приёмочных тест-сценариев (TS-01–TS-06) пройдены
- [ ] Время отклика < 2 сек для 95% операций при 20 одновременных пользователях
- [ ] Нет P0/P1 дефектов безопасности по OWASP Top 10
- [ ] Задокументированы: API (Swagger), ERD, руководство по развёртыванию
- [ ] Демо-стенд доступен по публичному HTTPS-адресу

### 6.2 Приёмочные тест-сценарии

| ID | Сценарий | Ожидаемый результат |
|---|---|---|
| **TS-01** | Создание и подтверждение номинации | PENDING → MATCHING → CONFIRMED после матчинга |
| **TS-02** | Блокировка при превышении лимита | Ошибка «Кредитный лимит исчерпан» |
| **TS-03** | Оформление Margin Call | Создано уведомление, зафиксировано в журнале |
| **TS-04** | Создание и оплата счёта | ВЫСТАВЛЕН → ОЖИДАЕТ → ОПЛАЧЕН |
| **TS-05** | Реноминация выше 110% | Предупреждение + требование одобрения |
| **TS-06** | Доступ по ролям (RBAC) | Роль «Биллинг» не видит модуль «Кредит» (HTTP 403) |

| Термин | Определение |
|---|---|
| **ETRM** | Energy Trading & Risk Management — класс систем автоматизации торговли энергоресурсами |
| **GTCP** | Gas Trading & Commercial Platform — наименование разрабатываемой системы |
| **MVP** | Minimum Viable Product — минимально жизнеспособный продукт |
| **Номинация** | Официальная заявка грузоотправителя о транспортировке объёма газа через точку в конкретные газовые сутки |
| **Реноминация** | Изменение объёма подтверждённой номинации в течение текущих газовых суток |
| **Матчинг** | Алгоритм сопоставления встречных номинаций ENTRY/EXIT |
| **ENTRY / EXIT** | Точки входа (Хоргош) и выхода (Господжинцы) газа в/из транспортной системы |
| **Margin Call (MCL)** | Официальное требование о пополнении обеспечения при превышении кредитного лимита |
| **Current Exposure** | Текущая финансовая задолженность контрагента перед компанией (EUR) |
| **FIRM** | Твёрдое бронирование мощности — не прерывается TSO в нормальных условиях |
| **INT** | Interruptible — прерывистое бронирование, может быть отозвано TSO |
| **TSO** | Transmission System Operator — оператор транспортной системы |
| **CAM NC** | Capacity Allocation Mechanisms Network Code — Европейский кодекс распределения мощностей |
| **Gas Day** | Газовые сутки: расчётный период с 06:00 до 06:00 CET следующего дня |
| **RBAC** | Role-Based Access Control — управление доступом по ролям |
| **JWT** | JSON Web Token — стандарт токенов для аутентификации REST API |
| **CET** | Central European Time — UTC+1, часовой пояс европейского газового рынка |
